# 대규모 스터디 참여형 RTC 서비스 아키텍처 설계 정리

## 1. 설계 목표

본 프로젝트는 **100명 이상의 사용자가 동시에 참여 가능한 실시간 스터디 환경**을 제공하는 것을 목표로 한다.

특히, SSAFY 프로젝트 환경의 제약(서버 비용, 인프라, 구현 난이도)을 고려하여  
다음과 같은 설계 원칙을 따른다.

- 실시간 상호작용 품질은 핵심 인원에 집중
- 네트워크 병목은 구조적으로 제거
- 모든 참여자는 “연결 상태”를 유지
- 영상 스트리밍은 최소화
- AI는 네트워크가 아닌 **클라이언트 로컬에서 활용**

---

## 2. 핵심 설계 개요

### 전체 구조 요약

- **발표자(1명)**  
  → WebRTC(OpenVidu)를 통해 실시간 영상 송출

- **일반 참여자(100명 이상)**  
  → 영상 송출 없음  
  → 로컬 AI로 얼굴 인식 수행  
  → 자리 있음(O) / 자리 비움(X) 상태만 서버로 전송  
  → 화면에는 상태에 따라 이미지 표시

즉,

> **영상 스트림은 항상 1개만 존재하고,  
> 나머지 사용자는 상태 기반 참여만 수행**

---

## 3. 전체 아키텍처 구조

[발표자 1명]
│
WebRTC Video (1 stream)
│
OpenVidu (SFU)
│
──────────────────────────
│
WebSocket (status only)
│
[참여자 N명]

- 로컬 카메라 ON

- 얼굴 인식 (AI)

- 자리 있음 / 비움 여부 판단

- 결과만 서버로 전송


---

## 4. 네트워크 설계 관점

### 4.1 연결(Connection)과 미디어(Media)의 분리

본 설계는 **연결 수와 서버 부하가 비례하지 않도록** 구성되어 있다.

| 항목 | 서버 부하 |
|----|----|
| 사용자 연결 유지 | 매우 낮음 |
| WebSocket 상태 메시지 | 매우 낮음 |
| 영상 스트림 1개 | 낮음 |
| 영상 스트림 다수 | ❌ 매우 높음 |

➡️ 서버 부하는 **연결 수가 아니라 “영상 스트림 수”에 의해 결정됨**

---

### 4.2 100명 이상 참여가 가능한 이유

- 100명 모두 연결 유지 가능
- WebRTC 영상 스트림: **항상 1개**
- 나머지 참여자는:
  - 영상 데이터 전송 ❌
  - 얼굴 인식 결과(boolean)만 전송 ⭕

예시 트래픽: 100명 × 1초 × 수십 바이트 ≈ 수 KB/s


➡️ **채팅 서버 수준의 부하**

---

## 5. 자리 비움(출석) 판단 로직

### 5.1 처리 위치

- 얼굴 인식 AI는 **클라이언트(브라우저) 로컬에서 수행**
- 서버는 영상 프레임을 수신하지 않음

### 5.2 판단 방식

- 카메라 프레임을 일정 주기로 분석
- 얼굴이 감지되면 → `자리 있음 (O)`
- 일정 시간 이상 감지되지 않으면 → `자리 비움 (X)`

서버로 전송되는 데이터 예시:
```json
{
  "userId": "user123",
  "present": true,
  "timestamp": 1710000000
}
```

## 6. UI 동작 방식

### 6.1 화면 구성

본 서비스의 화면은 **발표자 화면**과 **일반 참여자 상태 화면**으로 구분된다.

- **발표자**
  - WebRTC(OpenVidu)를 통한 실시간 영상 화면 표시
  - 항상 1명만 영상 송출 상태 유지

- **일반 참여자**
  - 실시간 영상 스트리밍 없음
  - 얼굴 인식 결과에 따라 상태 이미지 표시
    - `O` : 자리 있음 이미지
    - `X` : 자리 비움 이미지

### 6.2 UI 설계 특징

- 영상 스트리밍 없이도 참여 상태를 직관적으로 파악 가능
- 다수 참여자 환경에서도 UI 렌더링 부하 최소화
- 네트워크 트래픽과 UI 성능 간 완전한 분리

---

## 7. 발언자 전환 처리

### 7.1 기본 정책

- 동시에 활성화되는 영상 송출 인원은 **최대 1명**
- 발언자는 실시간으로 변경 가능
- 발언자 변경 시:
  1. 기존 발언자 영상 송출 OFF
  2. 신규 발언자 영상 송출 ON

➡️ 서버에는 항상 **1개의 영상 스트림만 존재**

---

### 7.2 동시 발언 처리 전략

여러 사용자가 동시에 발언하는 경우에도 영상 송출은 제한된다.

- 영상 송출 허용 인원: **1명**
- 우선순위 기준:
  - 음성 레벨(Active Speaker)
  - 손들기 기능
  - 최초 발언자 우선

다른 사용자는:
- 음성만 처리하거나
- 발언 대기 상태로 전환

---

## 8. 확장성 평가

본 설계는 대규모 참여 환경을 전제로 확장성을 평가하였다.

| 항목 | 평가 |
|----|----|
| 100명 이상 연결 | 가능 |
| 실시간 출석(자리 비움) 확인 | 가능 |
| 서버 비용 증가 | 매우 낮음 |
| 네트워크 병목 | 없음 |
| SSAFY 환경 적합성 | 매우 높음 |

➡️ 연결 수 증가가 서버 부하로 직결되지 않는 구조

---

## 9. 설계의 핵심 가치

본 설계의 핵심 가치는 다음과 같다.

- 영상 스트리밍을 최소화하여 네트워크 병목 제거
- WebRTC의 장점은 유지하고, 단점은 구조적으로 회피
- AI 처리를 서버가 아닌 **클라이언트 로컬**에서 수행
- 학생 프로젝트 환경에서도 구현 가능한 현실적 설계
- 향후 대규모 확장이 가능한 구조적 기반 확보

---

## 10. 결론

본 프로젝트는  
**100명 이상 참여 가능한 실시간 스터디 환경**을 목표로 하여,

- 영상 스트리밍은 발표자에게만 집중하고
- 일반 참여자는 상태 기반 참여 방식으로 전환함으로써
- 서버 비용, 네트워크 부하, 구현 난이도를 모두 고려한
  **현실적이고 확장 가능한 구조**를 설계하였다.

> 연결 수가 아니라  
> **동시에 존재하는 영상 스트림 수를 제어하는 것이  
> 대규모 RTC 설계의 핵심이다.**
