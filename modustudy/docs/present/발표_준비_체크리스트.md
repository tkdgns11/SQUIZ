## 예상 질문 & 답변

### 1. "AI 시스템 구조는 어떻게 되나요?"
> 하이브리드 방식입니다. 자체 파인튜닝한 Qwen3-8B로 1차 요약을 생성하고, Claude API로 검토/보완하여 최종 품질을 높입니다. 자체 모델만 사용하면 비용은 절감되지만 품질이 불안정할 수 있고, 외부 API만 사용하면 비용이 많이 듭니다. 하이브리드 방식으로 비용 효율성과 품질을 모두 확보했습니다. 스터디 템플릿 추천 같은 기능은 자체 AI만으로 처리합니다.

### 2. "AI 모델 파인튜닝은 어떻게 했나요?"
> 두 가지 모델을 파인튜닝했습니다.
>
> **Qwen3-8B (회의록 요약용)**
> - 데이터: GPT-4o-mini로 IT 스터디 회의록 요약 합성 데이터 생성
> - 학습: QLoRA (4-bit), LoRA r=8, alpha=8
> - 환경: SSAFY L40S 46GB GPU
> - 최종: GGUF Q4_K_M 양자화 (4.7GB)
>
> **Qwen3-14B (스터디 템플릿 추천용)**
> - 데이터: 스터디 계획 JSON 생성용 2,000개 (Train 1,800 / Val 200)
> - 학습: QLoRA (4-bit), LoRA r=32, alpha=64, 3 에폭
> - MAX_SEQ_LENGTH: 1536 (평균 토큰 1,031)
> - 결과: Train Loss 0.19, JSON 유효율 100%, 키 완전율 100%
> - 최종: GGUF Q4_K_M 양자화

### 3. "BLE 출석의 정확도는 어떤가요?"
> BLE 신호는 약 10m 이내에서 감지됩니다. 스터디장이 비콘을 발신하면 같은 공간에 있는 스터디원만 출석 체크되므로, 대리 출석을 방지할 수 있습니다. 또한 스터디장이 출석 상태를 수동으로 변경할 수 있어 오류 시에도 대응 가능합니다.

### 4. "WebRTC SFU 서버를 선택한 이유는?"
> P2P 방식은 참가자 수가 늘어날수록 각 클라이언트의 부하가 기하급수적으로 증가합니다. SFU(Selective Forwarding Unit) 방식은 서버가 중계 역할을 하여 클라이언트 부하를 줄이고, 다자간 미팅에서도 안정적인 품질을 제공합니다. mediasoup은 오픈소스이면서도 성능이 우수합니다.

### 5. "동시 접속자 수는 어느 정도 지원하나요?"
> 현재 아키텍처에서는 채팅 채널당 약 100명, 화상 미팅은 채널당 약 10-20명 정도를 권장합니다. 스케일 아웃이 필요한 경우 Redis Pub/Sub으로 서버 간 메시지를 동기화하고, SFU 서버를 추가 배포하는 방식으로 확장 가능합니다.

### 6. "Discord와 차별점은?"
> Discord는 범용 커뮤니케이션 툴이지만, Squiz는 스터디에 특화되어 있습니다.
> - AI 미팅 요약 및 회의록 자동 생성
> - BLE 기반 오프라인 출석 체크
> - 스터디 복습 퀴즈 AI 생성
> - 스터디 신청서/결과보고서 자동화
> - 게이미피케이션 (뱃지, 출석 스트릭, 작심삼일 패널티)
