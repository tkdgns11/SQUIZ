### 문제 #62: LEFT JOIN과 FULL OUTER JOIN의 차이점
**문제점**:
- 선택지 A와 B가 모두 정답으로 보이지만, 정답은 A만 지정됨
- 선택지 D는 명확히 틀렸지만, B도 정답으로 인정되어야 함

**현재 내용**:
```
질문: LEFT JOIN과 FULL OUTER JOIN의 차이점으로 올바른 것은?
선택지:
A. LEFT JOIN은 왼쪽 테이블의 모든 행을 반환한다.
B. FULL OUTER JOIN은 두 테이블의 모든 행을 반환한다.
C. LEFT JOIN은 오른쪽 테이블의 모든 행을 반환한다.
D. FULL OUTER JOIN은 중복된 행만 반환한다.
정답: A
```

**수정 제안**:
- 정답을 ["A", "B"]로 변경하거나, 질문을 "LEFT JOIN의 특징으로 올바른 것은?"으로 수정

### 문제 #116: JOIN의 종류가 아닌 것
**문제점**:
- "OUTER JOIN"을 오답으로 표시했으나, OUTER JOIN은 실제로 존재하는 JOIN 유형
- 해설에서 "LEFT, RIGHT, FULL이 포함됩니다"라고 했으나, 이들은 모두 OUTER JOIN의 하위 유형

**현재 내용**:
```
질문: 다음 중 JOIN의 종류가 아닌 것은 무엇인가요?
선택지: INNER JOIN, LEFT JOIN, OUTER JOIN, FULL JOIN
정답: ["C"]
해설: OUTER JOIN은 JOIN의 일반적인 용어이며, LEFT, RIGHT, FULL이 포함됩니다.
```

**수정 제안**:
- 해설을 "OUTER JOIN은 LEFT OUTER JOIN, RIGHT OUTER JOIN, FULL OUTER JOIN의 상위 개념이며, 단독으로 사용되지 않는 일반적 용어입니다"로 명확히 수정

```
질문: SQL에서 COALESCE 함수의 역할은 무엇인가요?

```

### 문제 #193-194: HAVING 절 관련 문제의 혼란
**문제점**:
- 문제 #220(번호 160)에서 "HAVING 절에서는 집계 함수의 별칭(total_revenue)을 직접 사용할 수 없습니다"라고 설명
- 실제로 일부 DBMS(MySQL 등)에서는 HAVING 절에서 별칭 사용 가능
- DBMS별 차이를 명시하지 않아 혼란 야기

**현재 내용**:
```
질문: 다음 쿼리에서 수정해야 하는 부분은?
HAVING total_revenue > 1000;
정답: C. total_revenue를 HAVING 절에서 사용할 수 없다.
해설: HAVING 절에서는 집계 함수의 별칭을 직접 사용할 수 없습니다.
```

**수정 제안**:
- 해설에 "표준 SQL에서는..." 또는 "PostgreSQL 등에서는..." 등 DBMS를 명시
- 또는 "대부분의 DBMS에서 권장되는 방법은 HAVING SUM(price) > 1000 형태로 작성하는 것입니다"로 수정

### 문제 #425: 트랜잭션 ACID 속성 (오류)

- C는 일관성(Consistency), 고립성(Isolation), 원자성(Atomicity), 지속성(Durability) 중 일관성만 해당
- 질문이 "'C'는 무엇을 의미하는가?"인데 4개를 모두 정답으로 표시한 것은 오류

**현재 내용**:
```

정답: ["A", "B", "C", "D"]
해설: ACID는 원자성, 일관성, 고립성, 지속성의 약자입니다.
```

**수정 제안**:
- 정답을 ["A"]로 수정 (일관성 = Consistency)
- 또는 질문을 "ACID 속성을 모두 고르세요"로 변경

### 문제 #568: LEFT JOIN과 WHERE 절 조합
**문제점**:
- "WHERE 절이 잘못됐다"를 정답으로 하지만, 해설이 모호함
- WHERE c.id IS NULL은 LEFT JOIN의 유효한 패턴 (outer exclusion)

**현재 내용**:
```
질문: SELECT * FROM orders o LEFT JOIN customers c ON o.customer_id = c.id WHERE c.id IS NULL;
에서 잘못된 부분은?
정답: B. WHERE 절이 잘못됐다.
해설: WHERE 절에서 c.id IS NULL은 LEFT JOIN의 결과에서 고객이 없는 주문을 찾는 것이 맞지만,
LEFT JOIN을 사용하는 이유를 무색하게 만들 수 있습니다.
```

**수정 제안**:
- 이 쿼리는 실제로 유효한 패턴이므로, 정답을 변경하거나 질문을 "비효율적인 부분은?"으로 수정

### 문제 #1194: 해시 충돌 해결 방법
**문제점**:
- "정렬"을 오답으로 표시했으나, 일부 해시 기법에서는 정렬을 활용함
- 리해싱을 충돌 해결 방법으로 분류하는 것은 논란의 여지가 있음

**현재 내용**:
```
질문: 해시 기반 자료구조에서 충돌 해결 방법으로 올바른 것을 모두 선택하세요.
```

## 패턴별 개선 제안

### 1. SHORT_ANSWER 유형
**문제점**:
- 정답 허용 범위가 불명확 (대소문자, 공백, 동의어 등)
- 일부 문제에서 인용 부호 포함 여부가 일관되지 않음

**개선 제안**:
- 정답 허용 패턴을 명시 (예: "DDL|Data Definition Language")
- 인용 부호가 포함된 문자열 답변의 경우 명확히 표시

### 2. MULTIPLE_CHOICE_MULTIPLE 유형
**문제점**:
- JSON 배열 형식 ["A", "B"]와 문자열 형식이 혼재
- 일부 문제에서 정답이 하나만 있는데도 복수 선택 형식 사용

**개선 제안**:
- 정답이 하나만 있으면 MULTIPLE_CHOICE로 변경
- JSON 배열 형식을 일관되게 유지

### 3. 코드 예제
**문제점**:
- 긴 SQL 쿼리나 코드가 한 줄로 표시되어 가독성 저하
- 일부 코드에서 특수 문자 이스케이프 문제

**개선 제안**:
- 코드 포맷팅을 개선하여 가독성 향상
- 특수 문자 이스케이프를 일관되게 처리

### 4. 해설
**문제점**:
- 일부 해설이 질문을 단순 반복
- 복잡한 개념에 대한 해설이 지나치게 간략

**개선 제안**:
- 해설에 "왜 그런지" 추가 설명
- 잘못된 선택지가 왜 틀렸는지도 간단히 언급
