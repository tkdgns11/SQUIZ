# Canvas 기반 WebRTC 영상 합성 구조 설명

본 문서는 WebRTC + OpenVidu 환경에서  
**화면공유와 캠을 클라이언트(Canvas)에서 합성하여  
하나의 영상 스트림으로 송출하는 구조**를 설명한다.

이 구조는 영상 스트림 개수를 1개로 유지함으로써  
100명 이상의 대규모 참여 환경에서도  
네트워크 안정성을 확보하기 위한 설계이다.

---

## 1. 전체 구조 개요

Canvas 합성 구조의 핵심은 다음과 같다.

- 화면공유와 캠을 **별도의 WebRTC 스트림으로 보내지 않음**
- 브라우저(Canvas)에서 두 영상을 **하나로 합성**
- 합성된 결과만 **단일 Video Track**으로 WebRTC(OpenVidu)에 전달

즉, 서버(OpenVidu)는  
“이 영상이 어떻게 만들어졌는지”는 전혀 알지 못하고  
**하나의 영상 스트림만 중계**한다.

## 3. 입력 단계: 두 개의 영상 소스

### 3.1 화면공유 (배경 영상)

- 브라우저 API: `getDisplayMedia()`
- 전체 화면 / 창 / 탭 중 선택
- Canvas에서 **배경 역할**

### 3.2 캠 영상 (오버레이)

- 브라우저 API: `getUserMedia()`
- 얼굴 영상
- Canvas에서 **작은 오버레이(PIP)** 로 사용

이 단계까지는 아직 WebRTC 전송이 시작되지 않는다.

---

## 4. 합성 단계: Canvas에서 수행되는 작업

Canvas는 일종의 “그림판” 역할을 한다.

매 프레임마다 다음 순서로 그림을 그린다.

1. Canvas 전체에 화면공유 영상을 그림
2. 캠 영상을 지정된 위치에 덮어 그림

예시 위치:
- 좌측하단
- 우측하단

캠 위치를 변경하는 경우:
- `drawImage()`의 좌표 값만 변경
- 새로운 스트림 생성 없음
- WebRTC 재협상 없음

즉,
> **캠 위치 변경 = Canvas 좌표 변경**

---

## 5. 출력 단계: Canvas → MediaStream

Canvas는 다음 API를 제공한다.

```js
canvas.captureStream(fps)
```

이 과정을 통해:

Canvas 전체가 하나의 영상이 됨

WebRTC 입장에서는 Video Track 1개

화면공유와 캠은 이 시점에서 이미 하나로 합쳐진 상태다.

## 6. WebRTC / OpenVidu 처리 방식

합성된 MediaStream은 OpenVidu Publisher를 통해 전송된다.

- Canvas에서 생성된 MediaStream은 단일 Video Track을 가진다
- OpenVidu 서버는 해당 스트림을 그대로 중계(SFU 방식)한다
- 서버가 인식하는 정보는 다음과 같다:
  - 영상 스트림 개수: 1개
  - 해상도: 720p
  - 프레임 레이트: 15~30fps

OpenVidu 서버는 다음 사항을 알지 못한다.

- 화면공유와 캠이 합성되었는지 여부
- 캠의 위치(좌측하단 / 우측하단)
- 클라이언트에서 수행된 영상 합성 과정

즉, 서버는 항상 하나의 영상 스트림만 처리한다.

---

## 7. 대규모 인원 수용이 가능한 이유

기존 방식에서는 화면공유와 캠을 각각 별도의 WebRTC 스트림으로 송출하였다.

- 화면공유 Video Track 1개
- 캠 Video Track 1개
- 총 2개의 영상 스트림을 참여자 수만큼 복제

Canvas 합성 구조에서는 다음과 같다.

- 최종 WebRTC 영상 스트림: 1개
- 서버 부하 계산식: 1 × (참여자 수 - 1)


따라서 참여자 수가 증가하더라도  
영상 스트림 개수는 증가하지 않으며  
네트워크 병목이 발생하지 않는다.

---

## 8. 캠 위치 반복 변경이 안전한 이유

캠 위치 변경은 다음과 같은 작업을 발생시키지 않는다.

- MediaStream 재생성 ❌
- WebRTC 재연결 ❌
- OpenVidu 서버 요청 ❌
- 네트워크 트래픽 증가 ❌

캠 위치 변경 시 실제로 수행되는 작업은 다음과 같다.

- Canvas 내부에서 `drawImage()` 좌표 변경
- 프론트엔드 렌더링 처리

따라서 버튼 클릭을 통한  
좌측하단 ↔ 우측하단 전환과 같은  
반복적인 위치 변경은 안정적으로 동작한다.

---

## 9. 안정성을 위한 필수 조건

본 구조의 안정성을 유지하기 위해  
다음 조건을 반드시 지킨다.

- 영상 합성은 클라이언트(Canvas)에서 수행
- WebRTC로 송출되는 영상 스트림은 항상 1개
- 최종 출력 해상도는 720p 기준
- 프레임 레이트는 15~30fps로 제한
- 추가적인 영상 스트림 송출 금지

---

## 10. 핵심 요약

- Canvas는 화면공유와 캠을 하나의 영상으로 합성하는 역할을 수행한다
- WebRTC는 합성된 영상을 실시간으로 전송한다
- OpenVidu는 단일 영상 스트림을 중계하는 SFU 역할을 담당한다



화면공유와 캠을 Canvas에서 하나의 영상으로 합성하여 송출하면  
서버는 항상 영상 스트림 1개만 처리하게 되며,  
이 구조를 통해 대규모 참여 환경에서도  
안정적인 실시간 서비스 제공이 가능하다.
